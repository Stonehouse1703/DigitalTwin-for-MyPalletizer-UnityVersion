== Quality Requirements

Die Qualitätsanforderungen machen die wichtigsten Ziele des Systems anhand konkreter Szenarien messbar.

=== Quality Requirements Overview

Für dieses System sind vor allem eine schnelle Reaktion, eine einfache Nutzung sowie ein robuster Betrieb wichtig. Da das System sowohl in der Lehre als auch für Tests verwendet wird, spielen zudem Portabilität und Wartbarkeit eine wichtige Rolle.

[cols="1,2", options="header"]
|===
| Qualitätsmerkmal | Priorität
| Performance | Sehr hoch
| Bedienbarkeit | Hoch
| Portabilität | Hoch
| Robustheit | Hoch
| Wartbarkeit | Mittel
|===

=== Quality Scenarios

[cols="1,2,3", options="header"]
|===
| ID | Qualitätsziel | Szenario

| **QS-1**
| Performance
| Ein Benutzer ruft `send_angles(...)` auf. Die Simulation soll innerhalb kurzer Zeit sichtbar auf den Befehl reagieren. Ziel ist, dass die Bewegung in Unity innerhalb von spätestens 50 ms beginnt.

| **QS-2**
| Portabilität
| Ein Benutzer wechselt von einem Windows-System auf macOS oder Linux. Dasselbe Python-Skript soll ohne Änderungen weiterverwendet werden können, sofern die benötigten Abhängigkeiten installiert sind.

| **QS-3**
| Robustheit
| Die Unity-Simulation wird während der Ausführung eines Python-Skripts beendet. Das Python-Skript soll dadurch nicht blockieren oder abstürzen.

| **QS-4**
| Eingabevalidierung
| Ein Benutzer sendet einen Gelenkwinkel ausserhalb des erlaubten Bereichs. Das System muss diesen Wert vor der Weitergabe an Simulation oder Hardware auf den zulässigen Bereich begrenzen.

| **QS-5**
| Bedienbarkeit
| Ein neuer Benutzer installiert die Software und führt ein Beispielskript aus. Innerhalb von 15 Minuten soll er in der Lage sein, eine Bewegung in der Simulation erfolgreich zu starten.

| **QS-6**
| Wartbarkeit
| Eine neue Nachricht für die Simulation soll ergänzt werden können, ohne bestehende Roboter- und Weltbefehle grundlegend umzubauen.

|===

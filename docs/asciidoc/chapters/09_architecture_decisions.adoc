== Architecture Decisions

Die folgenden Architectural Decision Records (ADR) beschreiben wichtige Architekturentscheidungen des Projekts.

=== ADR 1: Einheitliche Python-API für Simulation und Hardware

* **Kontext**: Der Benutzer soll nicht zwischen verschiedenen APIs für Simulation und reale Hardware unterscheiden müssen.
* **Entscheidung**: Die Klassen `Robot` und `World` bilden die zentrale Benutzerschnittstelle. Intern wird die eigentliche Ausführung an den `MyPalletizerController` oder an die Weltkommunikation delegiert.

* **Konsequenz**: Die Benutzung des Systems wird einfacher und konsistenter. Gleichzeitig übernimmt die Python-Seite mehr Verantwortung, da sie Validierung, Moduswahl und Weiterleitung der Befehle kapselt.

=== ADR 2: Kapselung der Herstellerbibliothek `pymycobot`

* **Kontext**: Der reale Roboter soll unterstützt werden, ohne dass der Benutzer direkt von der Herstellerbibliothek abhängig ist.
* **Entscheidung**: Die Bibliothek `pymycobot` wird nur intern im `MyPalletizerController` verwendet. Der Benutzer arbeitet ausschliesslich mit der eigenen `Robot`-Schnittstelle.

* **Konsequenz**: Externe Abhängigkeiten bleiben gekapselt und die API nach aussen bleibt stabiler. Dafür entsteht eine zusätzliche Abstraktionsschicht, die gepflegt werden muss.

=== ADR 3: Trennung von Simulation und realer Hardware

* **Kontext**: Dasselbe System soll sowohl mit der Simulation als auch mit dem realen Roboter funktionieren.
* **Entscheidung**: Die Simulation wird über UDP angesprochen, die reale Hardware über `pymycobot` und die serielle Verbindung. Die Auswahl erfolgt intern über die Modi `virtual`, `real` und `both`.

* **Konsequenz**: Dieselbe Python-API kann für Simulation, reale Hardware oder beide gleichzeitig verwendet werden. Dadurch bleibt die Benutzerschnittstelle einfach, obwohl intern unterschiedliche Kommunikationswege verwendet werden.


=== ADR 4: Trennung von Roboter- und Weltkommunikation

* **Kontext**: Das System verarbeitet zwei unterschiedliche Arten von Befehlen: Roboterbefehle und Befehle zur Verwaltung der Simulationswelt.
* **Entscheidung**: Diese beiden Bereiche werden über getrennte UDP-Kanäle umgesetzt. Port `5005` wird für die Robotersteuerung verwendet, Port `5006` für Weltbefehle.

* **Konsequenz**: Die Kommunikation bleibt klar getrennt. Häufige Bewegungsbefehle des Roboters werden nicht mit Weltbefehlen wie `spawn`, `delete` oder `clear` vermischt.

=== ADR 5: Verwendung von Presets für wiederholbare Szenarien

* **Kontext**: Typische Testszenarien sollen schnell geladen und reproduzierbar ausgeführt werden können.
* **Entscheidung**: Vordefinierte Szenarien werden als Presets auf Python-Seite bereitgestellt und über `World.load_preset(...)` geladen.

* **Konsequenz**: Tests und Demonstrationen werden einfacher und konsistenter. Gleichzeitig müssen Presets gepflegt werden, wenn sich die Simulationswelt ändert.

=== ADR 6: Verwendung von Articulation Bodies in Unity

* **Kontext**: Der Roboterarm besteht aus mehreren Gelenken, die in der Simulation stabil und nachvollziehbar bewegt werden müssen.
* **Entscheidung**: Für die Simulation der Gelenkkette werden in Unity `ArticulationBody`-Komponenten verwendet.
* **Konsequenz**: Die Abbildung des Roboterarms in Unity wird robuster und besser für zusammenhängende Gelenkstrukturen geeignet. Gleichzeitig wird die Konfiguration in Unity etwas aufwendiger.

== Runtime View

Die Laufzeitsicht zeigt, wie die einzelnen Bausteine des Systems zur Laufzeit zusammenarbeiten. Dabei ist wichtig, dass das System je nach Betriebsmodus (`virtual`, `real`, `both`) unterschiedliche Ausführungspfade verwendet. Die Python-Seite übernimmt die Orchestrierung, während die eigentliche Ausführung entweder in der Unity-Simulation, auf dem realen Roboter oder auf beiden Systemen gleichzeitig erfolgt.

=== Überblick

Die wichtigsten Laufzeitszenarien des Systems sind:

* asynchrone Roboterbewegung in der Simulation
* Roboterbewegung auf realer Hardware
* gleichzeitige Ausführung in Simulation und Hardware
* Verwaltung der Simulationswelt

Diese Szenarien werden in den folgenden Abschnitten beschrieben.

=== Szenario 1: Asynchrone Gelenkbewegung in der Simulation

Dieses Szenario beschreibt den typischen Ablauf, wenn der Benutzer einen Bewegungsbefehl im Modus `virtual` ausführt. Dabei wird ein neuer Soll-Zustand für alle vier Gelenke an die Unity-Simulation gesendet, ohne dass auf eine Rückmeldung gewartet wird.

[plantuml, target="diagram-runtime-virtual-async", format="png"]
----
@startuml
participant "Benutzer /\nPython-Skript" as User
participant "Robot" as Robot
participant "MyPalletizerController" as MPC
participant "protocol" as Protocol
participant "UDP Socket" as UDP
participant "UnityUdpBridge" as Bridge
participant "MyPalletizerAdapter" as Adapter
participant "Joint Actuators" as Joints

User -> Robot: send_angles(j1, j2, j3, j4, speed)
Robot -> MPC: send_angles(...)
MPC -> MPC: Winkel begrenzen
MPC -> MPC: Geschwindigkeit begrenzen
MPC -> MPC: _sim_angles aktualisieren
MPC -> Protocol: build_move_msg(...)
Protocol --> MPC: JSON-Payload
MPC -> UDP: sendto(payload, 127.0.0.1:5005)
UDP -> Bridge: UDP Paket empfangen
Bridge -> Adapter: move_joints(...)
Adapter -> Joints: Zielwinkel setzen
Joints -> Joints: Bewegung simulieren
@enduml
----

Ablauf:

1. Der Benutzer ruft `send_angles(...)` über die `Robot`-API auf.
2. `Robot` delegiert den Befehl an den `MyPalletizerController`.
3. Der Controller prüft die Werte, begrenzt Winkel und Geschwindigkeit und speichert die Zielwerte intern als Simulationszustand.
4. Anschliessend wird über das `protocol`-Modul eine JSON-Nachricht erzeugt.
5. Diese Nachricht wird per UDP an Port `5005` gesendet.
6. In Unity empfängt `UnityUdpBridge` das Paket und gibt die Daten an den Adapter weiter.
7. Der Adapter steuert die simulierten Gelenke an, wodurch die Bewegung in Unity sichtbar wird.

Dieses Szenario ist der Standardfall für flüssige Bewegungen in der Simulation.

=== Szenario 2: Bewegung auf realer Hardware

Dieses Szenario beschreibt den Ablauf im Modus `real`. Im Unterschied zur Simulation wird der Befehl nicht per UDP gesendet, sondern über die Herstellerbibliothek `pymycobot` an den realen Roboter weitergegeben.

[plantuml, target="diagram-runtime-real", format="png"]
----
@startuml
participant "Benutzer /\nPython-Skript" as User
participant "Robot" as Robot
participant "MyPalletizerController" as MPC
participant "pymycobot\nMyPalletizer260" as PMC
participant "USB / Serial" as Serial
participant "MyPalletizer 260 M5" as HW

User -> Robot: send_angles(j1, j2, j3, j4, speed)
Robot -> MPC: send_angles(...)
MPC -> MPC: Winkel begrenzen
MPC -> MPC: Geschwindigkeit begrenzen
MPC -> PMC: send_angles([j1,j2,j3,j4], speed)
PMC -> Serial: Serial-Kommando senden
Serial -> HW: Befehl übertragen
HW -> HW: Bewegung ausführen
@enduml
----

Ablauf:

1. Der Benutzer verwendet dieselbe API wie im Simulationsbetrieb.
2. `Robot` delegiert den Befehl an den `MyPalletizerController`.
3. Der Controller validiert die Eingaben.
4. Im Modus `real` wird der Befehl direkt an `pymycobot` weitergegeben.
5. Die Herstellerbibliothek sendet das Kommando über die serielle Verbindung an den realen Roboter.
6. Der Roboter führt die Bewegung physisch aus.

Dieses Szenario zeigt, dass dieselbe Benutzer-API auch für reale Hardware verwendet werden kann.

=== Szenario 3: Gleichzeitige Ausführung in Simulation und Hardware

Im Modus `both` wird derselbe Befehl gleichzeitig an die Simulation und an den realen Roboter gesendet. Dies ist die eigentliche Digital-Twin-Funktion des Systems.

[plantuml, target="diagram-runtime-both", format="png"]
----
@startuml
participant "Benutzer /\nPython-Skript" as User
participant "Robot" as Robot
participant "MyPalletizerController" as MPC
participant "protocol" as Protocol
participant "UDP Socket" as UDP
participant "UnityUdpBridge" as Bridge
participant "pymycobot\nMyPalletizer260" as PMC
participant "MyPalletizer 260 M5" as HW

User -> Robot: send_angles(j1, j2, j3, j4, speed)
Robot -> MPC: send_angles(...)

MPC -> MPC: Winkel begrenzen
MPC -> MPC: Geschwindigkeit begrenzen
MPC -> MPC: _sim_angles aktualisieren

par Simulation
  MPC -> Protocol: build_move_msg(...)
  Protocol --> MPC: JSON-Payload
  MPC -> UDP: sendto(payload, 127.0.0.1:5005)
  UDP -> Bridge: UDP Paket empfangen
else Hardware
  MPC -> PMC: send_angles([j1,j2,j3,j4], speed)
  PMC -> HW: Bewegung ausführen
end
@enduml
----

Ablauf:

1. Der Benutzer verwendet erneut dieselbe API.
2. Der `MyPalletizerController` erkennt, dass der Modus `both` aktiv ist.
3. Der Befehl wird parallel
   * an die Unity-Simulation per UDP und
   * an die Hardware über `pymycobot`
   weitergegeben.
4. Die Bewegung läuft dadurch gleichzeitig in der Simulation und auf dem realen Roboter.

Dieser Modus eignet sich besonders für Demonstrationen und für den direkten Vergleich zwischen Simulation und realem Verhalten.

=== Szenario 4: Verwaltung der Simulationswelt

Neben der Robotersteuerung unterstützt das System auch Änderungen an der Simulationswelt.

[plantuml, target="diagram-runtime-world", format="png"]
----
@startuml
participant "Benutzer /\nPython-Skript" as User
participant "World" as World
participant "protocol" as Protocol
participant "UDP Socket" as UDP
participant "WorldUdpBridge" as Bridge
participant "WorldSpawnManager" as WSM
participant "Unity Scene" as Scene

User -> World: spawn(obj)
World -> World: Objekt validieren
World -> Protocol: build_spawn_msg(...)
Protocol --> World: JSON-Payload
World -> UDP: sendto(payload, 127.0.0.1:5006)
UDP -> Bridge: UDP Paket empfangen
Bridge -> WSM: spawn / delete / clear
WSM -> Scene: Szene aktualisieren
@enduml
----

Ablauf:

1. Der Benutzer erzeugt ein Objekt und ruft `world.spawn(obj)` auf.
2. `World` prüft das Objekt und erzeugt eine passende JSON-Nachricht.
3. Die Nachricht wird per UDP an Port `5006` gesendet.
4. `WorldUdpBridge` empfängt die Nachricht und gibt sie an den `WorldSpawnManager` weiter.
5. Dieser aktualisiert die Unity-Szene und erzeugt, löscht oder leert Objekte.

Die Trennung zwischen Roboter- und Weltkanal ist wichtig, damit häufige Bewegungsbefehle und seltenere Weltänderungen unabhängig voneinander verarbeitet werden können.

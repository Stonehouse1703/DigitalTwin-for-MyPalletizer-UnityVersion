== Deployment View

Die Verteilungssicht beschreibt die physische Infrastruktur und die Softwareumgebung, in der das System betrieben wird.

Das System ist für den Betrieb auf einem normalen Rechner ausgelegt. Die Python-Anwendung und die Unity-Simulation laufen dabei in der Regel auf demselben Host. Optional kann zusätzlich der reale MyPalletizer 260 M5 über USB angeschlossen werden.

[plantuml, target="diagram-deployment-view", format="png"]
----
@startuml
node "Host PC\n(Windows / Linux / macOS)" as PC {
    component "Python Application" as PyApp
    component "Unity Application" as UnityApp
}

node "MyPalletizer 260 M5" as Robot

PyApp <--> UnityApp : UDP (localhost)\nPorts 5005 / 5006
PyApp --> Robot : USB (Serial)
@enduml
----

=== Host-PC

Der Host-PC benötigt folgende Softwarekomponenten:

* **Python Runtime (ab Version 3.12)**
  Erforderlich für die Ausführung der Steuerlogik und der Python-API.

* **Unity-Anwendung**
  Enthält die Simulation, die Visualisierung und die Empfängerlogik für die UDP-Kommunikation.

* **USB-Seriell-Verbindung**
  Wird benötigt, wenn der reale Roboter verwendet werden soll.

=== Bereitstellung

Die Software wird über eine CI/CD-Pipeline automatisiert gebaut. Dabei werden für die unterstützten Betriebssysteme jeweils Artefakte erstellt.

Dadurch kann der Benutzer die passende Version für sein Betriebssystem direkt herunterladen und starten, ohne die Anwendung selbst bauen zu müssen. Dies vereinfacht die Nutzung erheblich und ermöglicht einen schnellen Einstieg, insbesondere für Anwender die keine Unity-Vorkenntnisse haben.

Falls der Benutzer die Simulation anpassen oder erweitern möchte, bleibt das Unity-Projekt zugänglich. Es kann mit einer kostenlosen Unity-Version geöffnet und bearbeitet werden um die Simulation nach eigenen Bedürfnissen zu verändern.

=== MyPalletizer 260 M5

Der reale Roboter ist ein optionaler Bestandteil der Laufzeitumgebung. Er wird dann benötigt, wenn das System im Modus `real` oder `both` verwendet wird.

Für den Betrieb sind insbesondere folgende Punkte relevant:

* Verbindung über USB / Serial
* passende Firmware auf dem Roboter
* externe Stromversorgung für den zuverlässigen Betrieb

=== Betriebsarten

Das System kann in drei Varianten betrieben werden:

* **virtual** – nur Simulation
* **real** – nur realer Roboter
* **both** – Simulation und realer Roboter gleichzeitig

Diese Flexibilität ermöglicht es, das System an die jeweiligen Bedürfnisse und Ressourcen anzupassen. So kann beispielsweise auch ohne Zugriff auf die Hardware die Entwicklung und das Testen von Bewegungsabläufen in der Simulation erfolgen.

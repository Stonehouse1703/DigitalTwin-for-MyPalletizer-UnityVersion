[[section-concepts]]
== Cross-cutting Concepts

In diesem Kapitel werden die technischen Konzepte erläutert, die systemübergreifend implementiert wurden.

=== Physik-Konzept: Articulation Bodies

Die Simulation des MyPalletizer nutzt Unitys `ArticulationBody`-Komponente anstelle von Standard-Rigidbodies.

[plantuml, target="diagram-kinematic-chain", format="png"]
----
@startuml
skinparam componentStyle rectangle

package "Kinematische Kette (Unity Hierarchy)" {
    [Base (Root)] --> [J1 (Revolute)]
    [J1 (Revolute)] --> [J2 (Revolute)]
    [J2 (Revolute)] --> [J3 (Revolute)]
    [J3 (Revolute)] --> [J4 (Revolute)]
    [J4 (Revolute)] --> [End-Effektor (Gripper/Pump)]
}
@enduml
----

* **Vorteil**:
 Articulation Bodies verwenden "Reduced Coordinates" (Gelenkkoordinaten statt kartesischer Koordinaten). Dies verhindert das Auseinanderdriften der Gelenke bei schnellen Bewegungen oder Kollisionen, was bei Rigidbodies mit Joints oft ein Problem darstellt.
* **Steuerung (Drive Mode)**: Die Gelenke werden im `Targets`-Modus betrieben. Jedes Gelenk hat einen PID-Regler (Proportional-Integral-Derivative), der über die Parameter `Stiffness` (P) und `Damping` (D) konfiguriert wird.
* **Gelenk-Mapping**:
    - `j1`: Revolute (Basis-Rotation)
    - `j2`: Revolute (Unterarm)
    - `j3`: Revolute (Oberarm)
    - `j4`: Revolute (Handgelenk)
  Jeder `ArticulationJointActuator` übersetzt die eingehenden Winkel in den `target`-Wert des jeweiligen Drives.

=== End-Effektor-Konzept

Der MyPalletizer unterstützt verschiedene Werkzeuge, die in der Simulation funktional nachgebildet wurden:

[plantuml, target="diagram-pump-state", format="png"]
----
@startuml
[*] --> Idle
Idle --> Active : Pump-On Command
Active --> Attached : Trigger Enter & Object Found
Attached --> Releasing : Pump-Off Command
Releasing --> Idle : Destroy Joint Constraint
Attached --> Idle : Object Lost (Physics Reset)
@enduml
----

* **Greifer (Gripper)**:
  Wird über zwei gegenläufige `ArticulationBodies` simuliert. Ein `GripperController` übersetzt den binären Status (`open`/`close`) in Zielwinkel für die Greiferfinger. Physikalische Materialien mit hoher Reibung ermöglichen das Greifen von Objekten.
* **Saugpumpe (Suction Pump)**:
  Dieses Werkzeug nutzt ein physikalisches Constraint (z.B. ein `ConfigurableJoint`), das dynamisch erzeugt wird, wenn die Pumpe aktiviert ist und ein `GrabbableObject` den Trigger-Bereich berührt. Dies simuliert das "Ansaugen" realitätsnah.

=== Welt-Synchronisation & Objekt-Management

Um komplexe Szenarien reproduzierbar zu machen, implementiert das System ein ID-basiertes Objekt-Management.

* **Dynamisches Spawning**: Objekte erhalten beim Erzeugen in Python einen eindeutigen Namen. Dieser Name dient in Unity als Schlüssel, um das Objekt später gezielt zu verschieben oder zu löschen.
* **Prefab-System**: Unity hält eine Bibliothek von Prefabs (Box, Kugel, Zylinder) bereit. Der Python-Befehl definiert lediglich den Typ, die Position und die Skalierung.

=== Test-Konzept

Die Qualitätssicherung erfolgt auf drei Ebenen:

1. **Unit Tests (Python)**: Validierung der `protocol.py`. Testet, ob z.B. negative Geschwindigkeiten korrekt abgefangen oder auf den erlaubten Bereich begrenzt werden.
2. **Integration Tests (UDP)**: Ein automatisiertes Skript sendet Test-Patterns an Unity und prüft (via Log-Analyse), ob alle Pakete in der richtigen Reihenfolge verarbeitet wurden.
3. **Visuelle Validierung (Physik)**: Manuelle Tests zur Überprüfung der Kollisionserkennung. Beispiel: Kann der Roboter eine Box von A nach B heben, ohne dass diese durch den Greifer "rutscht"?

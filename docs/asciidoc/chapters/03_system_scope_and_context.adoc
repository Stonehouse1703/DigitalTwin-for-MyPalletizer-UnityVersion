== Context and Scope

In diesem Kapitel wird die Abgrenzung des Systems gegenüber externen Akteuren definiert sowie der operative Prozess (Workflow) dargestellt.

=== Business Context

Der Business Context beschreibt den Prozess aus Sicht der Anwender (Studierende/Dozierende). Das folgende Diagramm im BPMN-Stil visualisiert den typischen Ablauf einer Simulations-Session.

[plantuml, target="diagram-bpmn-process", format="png"]
----
@startuml
|Benutzer (Student)|
start
:Definiere Test-Szenario;
:Schreibe Python-Steuerlogik;
:Wähle Betriebsmodus\n(virtual / real / both);

|#LightBlue|Digital Twin System (Python API)|
:Starte Skript;
:Validiere Gelenklimits;

if (Modus?) then (virtual / both)
  :Serialisiere JSON;
  :Sende UDP (Port 5005/5006);
  |#LightGreen|Unity Simulation|
  :Empfange Paket;
  :Berechne Physik (ArticulationBody);
  :Visualisiere Bewegung;
else (real)
endif

if (Modus?) then (real / both)
  |#LightBlue|Digital Twin System (Python API)|
  :Sende Serial (USB);
  |#White|Physischer Roboter|
  :Bewege Servos;
else (virtual)
endif

|Benutzer (Student)|
:Analyse der Ergebnisse;
stop
@enduml
----

[cols="1,3", options="header"]
|===
| Akteur | Verantwortung im Prozess
| **Benutzer** | Erstellt die Programmlogik und bewertet das Verhalten des Roboters in der Simulation oder an der Hardware.
| **Python API** | Übernimmt die Orchestrierung, die Validierung der Parameter und die Kommunikation über die verschiedenen Protokolle.
| **Unity Simulation** | Bietet die visuelle und physikalische Validierung der geplanten Bewegungsabläufe.
| **Hardware** | Führt die validierten Befehle in der realen Welt aus.
|===

=== Technical Context

Der technische Kontext zeigt die Systemgrenzen und die spezifischen Kommunikationskanäle zwischen den Komponenten.

[plantuml, target="diagram-context-c4", format="png"]
----
@startuml
!include <c4/C4_Context.puml>

LAYOUT_WITH_LEGEND()

Person(user, "Student / Forscher", "Nutzt die Python API zur Robotersteuerung")
System_Boundary(dt_boundary, "Digital Twin Environment") {
    System(python_api, "MyPalletizer Python API", "Steuerungs-Hub & Hardware-Abstraktion")
    System(unity_sim, "Unity Simulation", "3D-Physik-Engine & Visualisierung")
}
System_Ext(robot, "MyPalletizer 260 M5", "Physischer Roboterarm")

Rel(user, python_api, "schreibt Code / nutzt", "Python")
Rel(python_api, unity_sim, "Steuerbefehle", "UDP / JSON (Port 5005/5006)")
Rel(python_api, robot, "Motorbefehle / LED", "Serial / USB")
@enduml
----

==== Externe Schnittstellen

[cols="1,1,2", options="header"]
|===
| Schnittstelle | Protokoll | Zweck
| **Benutzer-API** | Lokal (Python) | High-Level Funktionen wie `send_angles()`, `set_color()` oder `spawn_cube()`.
| **Hardware-Kanal** | Serial (115200 Baud) | Übertragung von Low-Level Befehlen an die M5Stack-Firmware des Roboters.
| **Simulations-Kanal** | UDP (v1 JSON) | Hochfrequente Übertragung von Soll-Zuständen an die Unity-Engine auf Port 5005.
| **Welt-Kanal** | UDP (v1 JSON) | Übertragung von Umgebungsbefehlen auf Port 5006.
|===

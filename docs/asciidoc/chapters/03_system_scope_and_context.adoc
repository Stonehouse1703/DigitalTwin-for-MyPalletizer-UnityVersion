== Context and Scope

In diesem Kapitel wird das System gegenüber externen Akteuren abgegrenzt und der typische Ablauf der Nutzung beschrieben.

=== Business Context

Der Business Context beschreibt den Ablauf aus Sicht der Benutzer (Studierende / Robotik-Interessierte).
Das folgende Diagramm zeigt eine typische Nutzung des Systems.

[plantuml, target="diagram-bpmn-process", format="png"]
----
@startuml
|Benutzer|
start
:Definiere Test-Szenario;
:Schreibe Python-Steuerlogik;
:Wähle Betriebsmodus\n(virtual / real / both);

|#LightBlue|Python API|
:Starte Skript;
:Validiere Eingaben;

if (Simulation aktiv?) then (ja)
  :Sende UDP (5005/5006);
  |#LightGreen|Unity Simulation|
  :Empfange Daten;
  :Simuliere Bewegung;
  :Visualisiere Ergebnis;
else (nein)
endif

if (Hardware aktiv?) then (ja)
  |#LightBlue|Python API|
  :Sende Befehle über Serial;
  |#White|Roboter|
  :Führe Bewegung aus;
else (nein)
endif

|Benutzer|
:Analyse der Ergebnisse;
stop
@enduml
----

[cols="1,3", options="header"]
|===
| Akteur | Verantwortung im Prozess

| **Benutzer**
| Erstellt die Programmlogik und bewertet das Verhalten des Roboters in der Simulation oder auf der Hardware.

| **Python API**
| Dient als zentrale Schnittstelle (Fassade) zum System.
Sie vereinfacht die Nutzung, prüft Eingaben und verteilt Befehle je nach Betriebsmodus an Simulation und/oder Hardware.

| **Unity Simulation**
| Stellt die Bewegung des Roboters visuell dar und ermöglicht das Testen von Szenarien.

| **Hardware**
| Führt die Befehle auf dem realen Roboter aus.

|===

=== Technical Context

Der technische Kontext zeigt die Systemgrenzen sowie die Kommunikationswege zwischen den einzelnen Komponenten.

[plantuml, target="diagram-context-c4", format="png"]
----
@startuml
!include <c4/C4_Context.puml>

LAYOUT_WITH_LEGEND()

Person(user, "Benutzer", "Nutzt die Python API zur Steuerung")

System_Boundary(dt_boundary, "Digital Twin Environment") {
    System(python_api, "Python API", "Zentrale Steuerung und Schnittstelle")
    System(unity_sim, "Unity Simulation", "Visualisierung und Simulation")
}

System_Ext(robot, "MyPalletizer 260 M5", "Physischer Roboter")

Rel(user, python_api, "nutzt", "Python")
Rel(python_api, unity_sim, "Steuerbefehle", "UDP (Ports 5005 / 5006)")
Rel(python_api, robot, "Steuerbefehle", "Serial (USB)")
@enduml
----

==== Externe Schnittstellen

[cols="1,1,2", options="header"]
|===
| Schnittstelle | Protokoll | Zweck

| **Benutzer-API**
| Lokal (Python)
| Stellt Funktionen wie `send_angles()`, `set_color()` oder `spawn()` zur Verfügung.

| **Hardware-Kanal**
| Serial (115200 Baud)
| Überträgt Steuerbefehle an den realen Roboter.

| **Simulations-Kanal**
| UDP (JSON)
| Überträgt Bewegungs- und Zustandsbefehle an die Simulation (Port 5005).

| **Welt-Kanal**
| UDP (JSON)
| Überträgt Befehle zur Verwaltung der Simulationsumgebung (Port 5006).

| **Trennung der Kanäle**
| -
| Die Kommunikation zur Simulation ist in zwei Kanäle aufgeteilt.
Der Roboterkanal ist für zeitkritische Bewegungsbefehle zuständig, während der Weltkanal für weniger häufige Aktionen wie das Erstellen oder Löschen von Objekten verwendet wird.

|===
